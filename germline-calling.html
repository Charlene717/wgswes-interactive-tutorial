STEP 7 / 16

生殖系變異呼叫

HaplotypeCaller 的局部重組裝、DeepVariant 的 CNN 影像、Strelka2 的貝氏速度——以及為何 gVCF 是聯合呼叫的關鍵。

HaplotypeCaller's local reassembly, DeepVariant's CNN images, Strelka2's Bayesian speed — and why gVCF is the key to joint genotyping.

概覽

為什麼 simple pileup 不夠？

早期變異呼叫器 (UnifiedGenotyper、bcftools call) 在 mismatch 位置直接看 pileup 投票。但 indel 周圍、重複序列、HLA 等位點，BWA 的對位本身就會出現系統性錯誤——這時你需要局部重組裝 (local de novo assembly)：丟掉原始對位，在窗口內重組讀數成 haplotype，再以 HMM 計算每條 read 對每條 haplotype 的概率。

2026 主流三家：HaplotypeCaller (GATK，局部組裝 + PairHMM)、DeepVariant (Google，CNN 看 pileup 影像)、Strelka2 (Illumina，貝氏 + indel 重組)。DRAGEN 整合自家 caller。

Early callers (UnifiedGenotyper, bcftools call) voted directly on pileups. But around indels, repeats, HLA, BWA's alignment is systematically wrong — you need local de novo assembly: discard the original alignment, reassemble reads in a window into haplotypes, then use HMM to compute the probability of each read given each haplotype.

The 2026 big three: HaplotypeCaller (GATK; local assembly + PairHMM), DeepVariant (Google; CNN over pileup images), Strelka2 (Illumina; Bayesian + indel reassembly). DRAGEN ships its own.

💡
gVCF 是聯合呼叫的必要條件：HaplotypeCaller 用 -ERC GVCF 為每樣本輸出 gVCF——既包含變異位點，也包含非變異區的「參考信賴度」區塊。沒有 gVCF，後續 GenomicsDB / CombineGVCFs / GenotypeGVCFs 都做不了。 gVCF is mandatory for joint calling: HaplotypeCaller with -ERC GVCF emits a per-sample gVCF — variants plus "reference confidence" blocks at non-variant sites. Without it, GenomicsDB / CombineGVCFs / GenotypeGVCFs can't run.

局部組裝

二、Active region 與 PairHMM

🎯

Active region

HC 先掃一遍 BAM，找出有 mismatch / indel 證據的窗口。其他區直接呼叫 ref，省 90% 計算。

HC scans the BAM and flags windows with mismatch/indel evidence. Other regions go straight to ref, saving 90% compute.

🧩

De Bruijn assembly

在 active region 內用 k-mer 圖重組裝 reads，得到候選 haplotypes (常見 2–10 條)。

K-mer De Bruijn graph reassembles reads in the window into candidate haplotypes (typically 2–10).

📐

PairHMM

每條 read × 每條 haplotype，HMM 計算 P(read | haplotype)。GATK 透過 SIMD/AVX 加速最慢的這一步。

For every read × every haplotype, PairHMM computes P(read | haplotype). The hot loop is SIMD/AVX-accelerated.

⚠️
downsampling 注意：HC 預設將 active region 內讀數 downsample 至 500×。極深 panel 必須 --max-reads-per-alignment-start 0 關閉，否則低 VAF 變異被丟。 Downsampling caveat: HC downsamples active-region reads to 500× by default. For ultra-deep panels, disable with --max-reads-per-alignment-start 0 or low-VAF variants get dropped.

三、Caller 抉擇

🌳 如何選 caller？

Q1:
需要 cohort 聯合呼叫？ → 是 → HaplotypeCaller -ERC GVCF + GenomicsDB + GenotypeGVCFs。
Q2:
追求最高精準度、有 GPU？ → DeepVariant (model 對應到你的 chemistry)。
Q3:
速度優先、單樣本？ → Strelka2 或 DRAGEN。
Q4:
PacBio HiFi 或 ONT 長讀？ → DeepVariant 長讀模型 或 Clair3。
Q1:
Need cohort joint calling? → Yes → HaplotypeCaller -ERC GVCF + GenomicsDB + GenotypeGVCFs.
Q2:
Top accuracy, GPU available? → DeepVariant with chemistry-matched model.
Q3:
Speed-first single sample? → Strelka2 or DRAGEN.
Q4:
PacBio HiFi or ONT? → DeepVariant long-read model or Clair3.
程式碼

三大 caller 實作

# HaplotypeCaller in GVCF mode (single-sample) / GVCF 模式
gatk HaplotypeCaller \
    -I bam/S1.recal.bam \
    -R ref/GRCh38.fa \
    -O gvcf/S1.g.vcf.gz \
    -ERC GVCF \
    --pair-hmm-implementation FASTEST_AVAILABLE \
    --native-pair-hmm-threads 4

# Optional: WES — restrict to capture intervals
gatk HaplotypeCaller -I bam/S1.recal.bam -R ref/GRCh38.fa \
    -L capture/targets.interval_list -ip 100 \
    -O gvcf/S1.g.vcf.gz -ERC GVCF
# DeepVariant via Singularity / Apptainer
BIN_VERSION="1.6.1"
singularity run docker://google/deepvariant:${BIN_VERSION} \
    /opt/deepvariant/bin/run_deepvariant \
    --model_type=WGS \
    --ref=ref/GRCh38.fa \
    --reads=bam/S1.recal.bam \
    --output_vcf=vcf/S1.dv.vcf.gz \
    --output_gvcf=gvcf/S1.dv.g.vcf.gz \
    --num_shards=16
# Strelka2 germline workflow
configureStrelkaGermlineWorkflow.py \
    --bam bam/S1.recal.bam \
    --referenceFasta ref/GRCh38.fa \
    --runDir strelka_S1/

strelka_S1/runWorkflow.py -m local -j 16
# Output: strelka_S1/results/variants/variants.vcf.gz

1. HaplotypeCaller 與 DeepVariant 在「如何決定變異」上的核心差別？

1. Core difference between HaplotypeCaller and DeepVariant in how they call variants?

A. 兩者完全相同A. They are identical
B. HC 用 CNN、DV 用 HMMB. HC uses CNN; DV uses HMM
C. HC 局部重組裝 + PairHMM；DV 把 pileup 轉成影像用 CNN 分類C. HC does local reassembly + PairHMM; DV converts pileups to images and classifies with a CNN
D. HC 只能跑 WES、DV 只能跑 WGSD. HC is WES-only; DV is WGS-only

2. 為什麼 cohort 聯合呼叫一定要 gVCF 而不是普通 VCF？

2. Why does cohort joint calling require gVCF instead of plain VCF?

A. gVCF 較小A. gVCF is smaller
B. gVCF 含「非變異位點的參考信賴度」，能區分「真 ref」與「未覆蓋」，聯合呼叫才有依據B. gVCF carries reference-confidence at non-variant sites, distinguishing true-ref from uncalled — essential for joint calling
C. 純粹是 GATK 規定C. Pure GATK convention
D. gVCF 不需索引D. gVCF needs no index

3. 是非題：HaplotypeCaller 的局部重組裝可用於 indel 周圍，但在純 SNV 區是浪費時間。

3. True/False: HC's local reassembly is useful around indels but wasted in pure-SNV regions.

A. 對A. True
B. 錯——active region 機制只在「有差異證據」的窗口才重組裝，純 ref 區直接跳過B. False — the active-region mechanism reassembles only in windows with evidence; pure-ref regions are skipped
